Introduction

Why Policy Operator? The GitOps way to manage Azure governance.

3 min read

What is Policy Operator?

Policy Operator is a Kubernetes-native controller that brings Azure Policy management into the GitOps world. It extends the Kubernetes API with custom resources — giving platform and security teams a declarative, version-controlled way to define, assign, and exempt Azure governance policies without ever leaving the tools they already know.

Instead of clicking through the Azure Portal or running imperative scripts, you write a YAML manifest, open a pull request, and let the operator reconcile the desired state against Azure’s Policy API.

The GitOps Way

Traditional Azure Policy management is imperative and fragile. You run a script, call an ARM template, or click through the portal — and the moment someone makes a change outside that process, drift is invisible until an audit catches it.

Policy Operator flips this model. Every policy definition, initiative, assignment, and exemption lives in Git as a Kubernetes manifest. Changes go through pull requests, code review, and CI pipelines before they ever reach Azure. The operator continuously reconciles — if something drifts in Azure, it is corrected automatically to match what Git says it should be.

The result: policy-as-code with real auditability, rollback via git revert, and zero manual portal access required.

Beyond Terraform and EPAC

Why not Terraform?

Terraform is a general-purpose infrastructure tool. Managing Azure Policy with it means wrapping policy JSON in HCL strings, losing syntax validation, and coupling governance lifecycles to infrastructure change cycles. Policy definitions buried inside Terraform modules are hard to review, test independently, or reuse across teams.

Policy Operator treats policies as first-class Kubernetes objects — with status conditions, events, and reconciliation loops — not as blobs of JSON glued into .tf files.

Why not EPAC?

Microsoft’s Enterprise Policy as Code (EPAC) is a powerful PowerShell-based framework, but it comes with significant operational overhead: a separate deployment pipeline, PowerShell expertise, its own configuration schema, and no native integration with Kubernetes-based platforms.

If your organisation already runs workloads on Kubernetes — whether on AKS or any other cluster — Policy Operator fits naturally into the same platform engineering workflow. You use the same RBAC, the same CI tooling, the same GitOps engine (Flux, Argo CD) that manages your application workloads. No separate toolchain to learn or maintain.

Extending Kubernetes

Policy Operator is not just a convenience wrapper. It is a genuine extension of the Kubernetes control plane into the Azure governance layer.

  • Custom Resources — AzurePolicyDefinition, AzurePolicyInitiative, AzurePolicyAssignment, AzurePolicyExemption are first-class Kubernetes objects with full kubectl support.
  • Status & Events — every resource reports its Azure sync status and surfaces errors as Kubernetes events, so kubectl describe gives you a full picture.
  • Controller reconciliation — the operator continuously watches both the Kubernetes API and Azure, healing drift automatically.
  • Platform composability — policy resources can be templated with Helm, generated by Crossplane, or composed into Platform Team abstractions — just like any other Kubernetes workload.

This makes Policy Operator the natural choice for teams building Internal Developer Platforms on Kubernetes who want Azure governance to be a first-class, auditable part of that platform.
